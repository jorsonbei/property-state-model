# PSM / 物性AI完整复盘与执行路线图 V0.248-V0.260

日期：2026-07-14

## 一、当前结论

当前项目的真实位置是：

- 已完成一个可运行的 `Property-State Model` 外骨骼。
- 已建立 `Q -> Omega -> phi -> Delta sigma -> Pi -> eta -> B_sigma -> Sigma+` 的确定性状态路由、候选审计、控制器重写、失败账本和版本化验证链。
- 已有正常聊天外壳和本地 Ollama 回答路径。
- 尚未形成经过独立真实数据验证的物性原生基础模型。
- 当前最强证据证明的是规则系统在自建案例上的一致性和安全边界，不是开放域聊天智能或真实世界泛化能力。

因此，下一阶段不再以继续堆叠同构安全案例为主要目标。主线改为：先修复项目真相和聊天质量，再引入真实外部裁判与独立数据，最后进入可训练状态模型。

## 三、架构实现度

| 物性层 | 当前实现 | 真实完成度 | 主要缺口 |
|---|---|---:|---|
| `phi_state` | 关键词和固定模板抽取 | 原型 | 不能稳定处理隐含语义、多对象和上下文变化 |
| `Q` | 领域规则、否决和复核状态 | 可用原型 | 规则与配置分散，缺少真实任务证据绑定 |
| `Omega` | 按风险等级产生路线标签 | 原型 | 只给路线名，没有真正调用检索、工具、沙盒或专家裁判 |
| `Delta sigma` | 关键词压力识别 | 原型 | 缺少数值、时间变化和多尺度状态建模 |
| `Pi` | 固定参与者、产物和依赖模板 | 骨架 | 尚未形成任务级动态关系图 |
| `eta` | 固定未知项和长尾提示 | 骨架 | 没有从数据和失败账本学习异常分布 |
| `B_sigma` | 词法候选审计和确定性重写 | 边界可用 | 能拦强承诺，但不能识别事实幻觉、答非所问和虚构路线图 |
| `Sigma+` | 审计报告、候选门控和失败账本 | 内部可用 | 尚未把真实外部验证结果接入最终自然回答 |
| 状态编码器 | 透明规则/签名编码器 | 研究基线 | 不是神经基础模型，当前 1.000 指标不能代表开放域泛化 |
| 聊天产品 | 本地网页、API、Ollama 回答 | 可体验原型 | 回答质量、多轮记忆、事实接地和加载体验未达产品门槛 |

## 四、已确认的问题

### P0：执行前必须修复

1. `CURRENT_STATUS.md` 和根 `README.md` 采用历史内容不断嵌套追加的方式。顶部是 V0.247，但文件尾部仍含旧版本“下一步 V0.72”等冲突信息。
2. V0.248 之后尚未定义 V0.249 family。若直接运行当前推进脚本，`next_family=None` 会回退到旧的 V0.231 计划，导致下一阶段状态错误。
3. 回归工具默认目录为 `fixtures_out`，V0.247 实际使用 `fixture_out`，路径契约已经漂移。
4. 多轮聊天服务器只保留用户消息，忽略助手消息，却把 `context_carried_forward` 标为真。
5. 多轮审计文本注入 `Q 核` 等术语，反过来污染领域识别，使普通追问容易被路由成 `wuxing_theory/high`。

### P1：聊天质量主线

1. 项目状态和开发计划问题交给 4B 本地模型自由生成，已实测出现虚构功能、虚构时间表和泛化客服话术。
2. `物性AI` 不是状态抽取器的物性领域关键词，相关问题会被标成 `general`。
3. 当前审计器主要检查强承诺、边界擦除和假引用，无法检查答非所问、事实不接地、重复模板或虚构项目状态。
4. 高风险 fallback 会重复用户问题并输出固定三步模板，可能安全但没有实际帮助。
5. 前端等待模型时没有主聊天区加载状态；本地回答实测可超过 11 秒。
6. 当前所谓多轮上下文没有形成“用户消息 + 助手回答 + 状态摘要”的闭环。

### P2：研究证据与工程维护

1. 训练/验证/测试切分按领域和风险排序，长期固定为 2172/10/10；后续 2000 多条新案例几乎全部进入训练集。
2. 测试集来自最早的 10 个案例，没有按 family、来源或时间冻结，不能测量新 family 的泛化。
3. 大多数案例由同一规则模板生成，标签也来自同一规则链，1.000 指标存在同源闭环。
4. `psm_v0.__init__.VERSION` 仍为 `psm_v0.20`，运行时状态包版本与项目状态版本分离且未显式说明。
5. 根目录 YAML、schema 和检查清单没有成为运行时单一配置源，代码中存在重复常量。
6. `project_status_exporter.py`、`optional_external_release_summary.py` 和推进脚本已经过度膨胀，版本逻辑难以维护。
7. 产物重复保存造成约 5.9 GB 占用，尚无保留策略、清单哈希和历史归档层。
8. 当前工作目录不是 Git repository；产物虽有版本号，代码本身缺少提交级变更历史和可靠回滚点。

## 五、执行路线

### V0.248：恢复点与工具链硬化

目标：确保版本恢复、推进和回归只有一个真相源。

施工：

- 以结构化 `project_status` JSON 为唯一机器真相源。
- 把 `CURRENT_STATUS.md` 改成短当前摘要；历史内容转入独立 archive/index，不再递归嵌套。
- 统一 `fixture_out` 路径和回归参数。
- 在运行 optional refresh 前定义 V0.249 family，删除无关的 V0.231 fallback。
- 明确 runtime schema version、project version、model version 三者含义。
- 生成只读产物清单和空间报告；暂不删除历史数据。
- 建立本地、非破坏性的版本控制基线；不配置远端、不上传数据。

验收：

- 状态文件中只能出现一个 current version 和一个 next version。
- V0.247 回归复跑通过。
- V0.248 optional refresh 通过，raw/gated 风险保持 0/0。
- V0.248 的 next stage 必须准确指向 V0.249。

### V0.249：聊天意图、接地和多轮修复

目标：先让“我问，你回答”在语义上成立。

施工：

- 对话历史保留 user/assistant 角色，不再丢弃助手回答。
- 对话语义上下文与内部审计上下文分离，PSM 注入词不得改变用户原始领域。
- 增加 `物性AI`、项目状态、路线图、身份、能力、理论解释等意图。
- 项目状态和路线图回答必须读取本地结构化事实，禁止由生成模型编造。
- 新增 relevance/grounding 审计：直接性、问题覆盖、事实来源、非回避、非重复模板。
- 把本轮已复现的失败案例纳入回归。

验收：

- “项目做到哪里”必须回答 V0.247/V0.248 的真实状态，不得虚构未来月份。
- “刚才第二阶段叫什么”必须读取助手上一轮内容并正确回答。
- 普通多轮追问不得因隐藏 PSM 术语被误判为高风险物性任务。
- 内容问题不得回复“页面已经是正常聊天模式”。

### V0.250：回答模型对照与生成契约

目标：确定本地可用模型组合，而不是凭模型名称切换。

施工：

- 建立 `gemma3:4b` 与 `deepseek-r1:8B` 的同题盲评。
- 测量直接性、事实接地、中文自然度、边界帮助度、延迟和失败率。
- 将回答生成接口抽象为 provider，保留确定性 fallback。
- 生成结果至少区分 `answer`、`grounded_facts`、`uncertainties`、`required_judges`，正文仍保持自然聊天。

验收：

- 冻结模型选择报告和基准集。
- 任何模型不得在项目状态题上绕过结构化接地层。
- 普通问题中位响应目标小于 5 秒；超时必须有可理解的降级回答。

### V0.251：独立聊天黄金集

目标：建立不由现有规则模板反推的质量证据。

施工：

- 建立至少 80 条人工设计的真实问题，覆盖闲聊、解释、物性论、项目状态、写作、代码、研究、交易、医疗和法律。
- 按来源/family 冻结 train/dev/blind，不能按记录排序切分。
- 设立最少 20 条不可回流的 blind chat set。
- 评价正确性、相关性、可执行性、边界质量、多轮一致性和幻觉。

验收：

- 通过独立 blind gate；安全通过和回答有用性分别报告。
- 不再用单一“risk=0”代表聊天质量。

### V0.252：产品交互与浏览器回归

目标：让本地体验达到稳定内部 alpha。

施工：

- 增加生成中状态、取消、超时、重试和错误恢复。
- 支持逐步显示或流式回答。
- 调试信息默认隐藏，且永远不进入普通回答消息。
- 增加桌面和移动视口的自动化回归、键盘操作和基本无障碍检查。

验收：

- 无重复消息、无布局溢出、无主聊天调试泄漏。
- 失败时保留用户输入并给出可重试状态。

### V0.253：真实 Omega 路由与外部裁判

目标：从“给路线标签”升级为“真的执行路线”。

施工：

- 项目问题连接本地状态读取器。
- 事实问题连接来源/检索适配器。
- 代码问题连接静态检查、测试和沙盒适配器。
- 文件问题连接真实文件读取与证据引用。
- 每个工具结果写入 provenance 和 failure ledger。

验收：

- 至少四类路线能产生真实工具证据。
- 工具失败不能被语言覆盖，必须降级并入账。

### V0.254：动态 Pi、eta 与失败学习

目标：让关系腔和异常项来自任务，而不是固定模板。

施工：

- 从消息、文件、工具和裁判结果构建任务依赖图。
- 区分已知事实、推断、未知、冲突和待验证项。
- 从失败账本生成回归候选，但必须经过独立筛选后才能进入训练。

验收：

- 同一问题加入新证据后，状态图和下一步协议可解释地变化。
- 不允许失败案例自动回流污染 blind set。

### V0.255：内部聊天 Alpha 门

目标：决定是否允许稳定内部日常使用。

硬门：

- V0.251 blind chat gate 通过。
- 多轮记忆、项目接地、普通聊天和高风险帮助式边界通过。
- 浏览器与 API 回归通过。
- 关键事实幻觉为 0；critical safety false negative 为 0。
- 外部用户试用仍保持关闭。

### V0.256-V0.258：可训练物性状态模型

目标：开始真正的 V1 状态编码器研究。

施工：

- 建立与规则模板来源隔离的状态标注协议。
- 为 `Q/Omega/phi/Delta sigma/Pi/eta/B_sigma` 定义结构化目标和标注者分歧。
- 训练可替换的状态编码候选，与规则基线和普通 LLM 基线比较。
- 使用 family/source/time holdout，禁止同源模板跨 split。
- 只有独立集提升且 critical false negative 不增加时，才允许进入 shadow。

验收：

- 指标必须分成训练内、开发集、冻结盲集和真实任务集。
- 通过 admission 不等于替换规则；先保持 shadow-only。

### V0.259：Sigma+ 交付闭环

目标：把自然回答、状态、证据和失败统一成可追溯交付。

施工：

- 普通用户看到自然回答。
- 研发可展开查看状态链、来源、工具结果、失败与声明等级。
- 每条强结论可追溯到证据或明确标为假设。

验收：

- 随机抽样强结论全部有 provenance 或降级标记。
- 自然回答不暴露不必要的内部术语。

### V0.260：内部试用就绪评审

目标：完成内部真实使用评审，不自动开放外部试用。

交付：

- 安全报告、聊天质量报告、盲测报告、模型对照报告、性能报告、失败账本摘要和剩余风险。
- 明确结论只能是 `internal_trial_ready`、`needs_more_work` 或 `blocked`。
- 外部用户试用、隐私合规、账号系统、公开服务和高风险专业用途需要独立的新阶段授权。

## 六、执行顺序与停止条件

下一条可执行任务是 V0.248，但必须先完成 P0 工具链修复，再运行 optional refresh。

默认持续执行顺序：

```text
恢复点硬化
-> V0.248 external refresh
-> V0.249 chat grounding/memory
-> V0.250 model bakeoff
-> V0.251 independent blind chat gate
-> V0.252 product UX
-> V0.253 real judges
-> V0.254 dynamic state
-> V0.255 internal alpha gate
-> V0.256-V0.260 trainable PSM research and internal review
```

只有以下情况需要用户介入：

- 是否引入需要付费或上传数据的外部模型/API。
- 是否允许真实外部用户进入试用。
- 是否处理可能包含隐私、医疗、法律或交易真实数据的评测集。
- 是否删除或迁移历史产物以释放磁盘空间。

在这些情况出现前，本地代码、数据构建、验证、文档和服务器维护由 Codex 按本路线继续完成。
